01 Database ERD

Three tables. Every holding and every trade traces back to a single user, enforced at the schema level with a unique constraint on (user_id, symbol).

Primary / foreign key field Relationship, cardinality labelled 1 / N
On buy

Holding is upserted and avg_buy_price recalculated as a weighted average; a row is appended to Transactions; cash_balance is debited.

On sell

Holding quantity decreases and the row is deleted at zero; a row is appended to Transactions; cash_balance is credited.

Why one row per symbol

The UNIQUE(user_id, symbol) constraint keeps average-cost accounting correct — a user can hold AAPL only once, never as scattered lots.

02 User Flow

From an anonymous visitor to a recorded trade. The buy/sell fork is the only branch — both sides reconverge into the same portfolio and history views.

Buy path Sell path Dashed = data refresh, not a page navigation

03 System Architecture

The browser never talks to Finnhub. Every price quote is relayed through the API so the key never leaves the server.

Auth

Stateless JWT, 7-day expiry, validated on every request to /api/portfolio and /api/transactions.

Secrets

Finnhub key and DB connection string live in .NET User Secrets locally, host environment variables in production — never in source control.

Caching

Quote responses cached briefly server-side to stay well under Finnhub's 60 calls/min free-tier limit.
